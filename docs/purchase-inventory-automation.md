# Purchase + Inventory — Automated Working Process

End-to-end procure-to-pay (P2P) and inventory flow after automation. The goal
is to make staff work with decisions, not data entry: every document code is
server-generated, one-click hand-offs carry data forward, and GRN posting updates
inventory stock atomically.

---

## 1. The big picture

```
                          ┌──────────────────────┐
                          │   SUPPLIER MASTER     │  SUP-0001  (auto)
                          │  (vendors, ACTIVE)    │
                          └──────────┬───────────┘
                                     │ feeds the "Supplier" dropdown everywhere
                                     ▼
  ┌─────────┐   Raise RFQ   ┌──────────┐  Convert to PO  ┌──────────┐
  │   PR    │ ────────────► │ SOURCING │ ─────────────►  │    PO    │
  │ PR-0001 │               │ RFQ-0001 │                 │ PO-0001  │
  └────┬────┘               └──────────┘                 └────┬─────┘
       │  Convert to PO (repeat / direct buy)                 │
       └─────────────────────────────────────────────────────┤  Receive (GRN)
                                                              ▼
                                                        ┌──────────┐
                                                        │   GRN    │  GRN-0001
                                                        │ receipt  │
                                                        └────┬─────┘
                                         Post to inventory   │      Raise Payment
                              ┌──────────────────────────────┤
                              ▼                               ▼
                     ┌─────────────────┐              ┌──────────────┐
                     │ STORE INVENTORY │              │   PAYMENT    │  PAY-0001
                     │  STK-0001 stock+│              │  REQUESTED   │
                     └─────────────────┘              └──────────────┘
```

All document numbers above are generated by the server, never typed by users.

## 2. Numbering rules (system-generated, locked)

| Document            | Prefix | Example   | Scope |
|---------------------|--------|-----------|-----------------------------------------|
| Supplier Master     | `SUP`  | SUP-0001  | per organization                        |
| Purchase Requisition| `PR`   | PR-0001   | per organization                        |
| Sourcing / RFQ      | `RFQ`  | RFQ-0001  | per organization                        |
| Purchase Order      | `PO`   | PO-0001   | per organization                        |
| Goods Receipt Note  | `GRN`  | GRN-0001  | per organization                        |
| Payment Request     | `PAY`  | PAY-0001  | per organization                        |
| Store Inventory     | `STK`  | STK-0001  | per organization                        |
| Machine Inventory   | `MCH`  | MCH-0001  | per organization                        |
| Metal Inventory     | `MTL`  | MTL-0001  | per organization                        |

Key behavior:
- Format is always `PREFIX-0001` with a 4-digit numeric suffix.
- Each organization has independent counters.
- Counters are seeded from the highest existing matching code in the database.
- Code generation is atomic and transaction-safe.
- Reserved fields like `docNo` / `itemCode` are stripped from all incoming
  client payloads.
- Codes are read-only in the UI and are set only by server-side handlers.

## 3. Workflow overview

### Step 0 — Supplier Master
- Create suppliers via **Purchase → Suppliers → New supplier**.
- `SUP-` codes are generated on save.
- Only suppliers with `Status = ACTIVE` appear in dropdowns for PR/RFQ/PO/GRN/Payment.

### Step 1 — Purchase Requisition (PR)
- Open **Purchase → Requisition → New requisition**.
- `PR-` code appears after save.
- `Requested By` and `Department` are populated from the authenticated user,
  not from client input.
- Enter `itemName`, `quantity`, `uom`, and required dates.
- The UI can detect repeat purchases and suggest the prior supplier + last rate.
- PR progresses through Draft → PR Raised → Production Approval → Approved.

### Step 2 — PR hand-off
From an approved PR:
- **Raise RFQ** opens a sourcing form with `prRef`, item, qty, UOM prefilled.
- **Convert to PO** opens a PO form with `rfqRef = PR No.`, supplier, item, qty,
  rate, and amount prefilled.
- The PR source status is advanced by the promotion action.

### Step 3 — Sourcing / RFQ
- RFQ inherits PR fields and is used for supplier negotiation.
- Once ready, **Convert to PO** creates a new PO prefilled from the RFQ.
- The RFQ source status becomes `SELECTED`.

### Step 4 — Purchase Order (PO)
- Create a PO with an auto-generated `PO-` code.
- After approval and send, **Receive (GRN)** opens a GRN form prefilled with
  a PO line.
- The new GRN captures `poRef`, `itemName`, ordered qty, and default received qty.

### Step 5 — Goods Receipt Note (GRN)
- Create a GRN with `GRN-` code generated by the server.
- Choose the arrival warehouse and record inspection details.
- Received quantities are entered per invoice/PO line.
- Receipt status is calculated automatically from received quantities.

### Step 6 — Post stock to inventory
- From the GRN, click **Post to inventory**.
- The server updates inventory in one transaction.
- For each received item:
  - If a store item exists with the same `itemName` (case-insensitive), its
    `currentStock` is incremented.
  - Otherwise a new store item is created with an `STK-` code.
- The GRN is marked `stockUpdated = YES` and status `STOCK_UPDATED`.
- Re-posting the same GRN is a no-op.

### Step 7 — Payment Request
- From the GRN preview, **Raise Payment** opens a payment request with:
  - `supplier`
  - `poRef`
  - `invoiceNo`
  - total `invoiceAmount` from GRN line amounts
- Payment requests get auto-generated `PAY-` codes.

## 4. Real case examples

### Case 1: Repeat/direct buy from PR to PO
1. A planner creates a new requisition for `Steel Rods`:
   - `itemName`: Steel Rod
   - `quantity`: 100
   - `uom`: PCS
   - `requiredBy`: 2026-06-15
   - The system saves it as `PR-0005`.
2. The planner sees the item is a repeat purchase and the suggested supplier
   is `ABC Metals` with last rate `50.00`.
3. The planner clicks **Convert to PO**.
4. The new purchase order is opened with:
   - `rfqRef = PR-0005`
   - `supplier = ABC Metals`
   - `itemName = Steel Rod`
   - `quantity = 100`
   - `uom = PCS`
   - `rate = 50.00`
   - `amount = 5,000.00`
   - status `DRAFT`
5. When the PO is saved, the server issues `PO-0006`.

### Case 2: New item sourcing flow (PR → RFQ → PO)
1. A buyer creates a requisition for `Hydraulic Filter`:
   - `itemName`: Hydraulic Filter
   - `quantity`: 20
   - `uom`: NOS
   - The system saves it as `PR-0010`.
2. The buyer clicks **Raise RFQ**.
3. The RFQ opens pre-filled with:
   - `prRef = PR-0010`
   - `itemName = Hydraulic Filter`
   - `quantity = 20`
   - `uom = NOS`
   - `status = SOURCING`
4. The buyer records supplier quotes and selects `XYZ Parts Co.`.
5. The buyer clicks **Convert to PO**.
6. The PO is created with:
   - `rfqRef = RFQ-0011`
   - `supplier = XYZ Parts Co.`
   - `itemName = Hydraulic Filter`
   - `quantity = 20`
   - `uom = NOS`
   - `rate = 120.00`
   - `amount = 2,400.00`
   - status `DRAFT`
7. When saved, the server issues `PO-0012`.

### Case 3: GRN receipt and inventory posting
1. A warehouse clerk receives goods for `PO-0012` and creates a GRN.
2. The GRN opens as a new record with code `GRN-0013`.
3. The GRN line is seeded from the PO and contains:
   - `poRef = PO-0012`
   - `itemName = Hydraulic Filter`
   - `invoiceQty = 20`
   - `receivedQty = 20`
   - `amount = 2,400.00`
4. The clerk enters `invoiceNo = INV-5001` and saves the GRN.
5. The clerk clicks **Post to inventory**.
6. The server aggregates received quantities and updates stock:
   - If store inventory already has `itemName = Hydraulic Filter`, its
     `currentStock` is incremented by `20`.
   - If no matching store item exists, a new item is created as `STK-0003`
     with `currentStock = 20`, `warehouse` copied from the GRN, and `unitRate = 120.00`.
7. The GRN is marked `stockUpdated = YES` and its status becomes `STOCK_UPDATED`.
8. Re-clicking **Post to inventory** again does nothing; the operation returns
   `alreadyPosted: true`.

### Case 4: Payment request from GRN
1. After posting stock, the user clicks **Raise Payment** on `GRN-0013`.
2. The payment form opens with:
   - `supplier = XYZ Parts Co.`
   - `poRef = PO-0012`
   - `invoiceNo = INV-5001`
   - `invoiceAmount = 2,400.00`
3. When saved, the system creates `PAY-0004`.

## 5. Edge cases and implementation details

### Document codes and counter seeding
- Code generation uses `lib/sequence/next-code.ts`.
- Each scope is keyed by `pur:<orgId>:<submodule>` or `inv:<orgId>:<submodule>`.
- The first code in a scope is seeded from the highest existing code matching the
  prefix in the database.
- This preserves imported/demo data such as `PO-0005` and avoids collisions.
- If the scope counter row does not exist, it is created with `upsert` in the
  same transaction.
- The entire operation is transaction-safe: if create fails, the code is not
  permanently consumed.

### Reserved fields are ignored
- Incoming client payloads may include reserved keys like `docNo` or `itemCode`.
- The server strips these before create/update.
- The system always generates these values itself.

### PR user fields are authoritative
- PR fields flagged with `prefillUser` are replaced by real user identity on save.
- This prevents clients from spoofing `Requested By` or `Department`.

### Promotion mapping rules
- `PR → RFQ`: carries `prRef`, supplier, item, qty, UOM, quotedRate.
- `PR → PO`: carries `rfqRef = PR No.`, supplier, item, qty, rate, amount.
- `RFQ → PO`: carries `rfqRef = RFQ No.`, supplier, item, qty, rate, paymentTerms.
- `PO → GRN`: seeds one invoice line with `invoiceQty = receivedQty = PO qty`.
- `GRN → Payment`: rolls up the first `invoiceNo` and the first `poRef`, plus
  the total amount from all GRN line items.
- The source document’s status is advanced on promotion when configured.

### GRN stock posting
- The server aggregates all `receivedQty` values across GRN invoice lines.
- Items with empty names or non-positive quantities are ignored.
- If the GRN has no valid received quantities, the operation fails with:
  `This GRN has no received quantities to post.`
- Matching store inventory is case-insensitive by `itemName`.
- If multiple store items share the same name, the oldest created match wins.
- Existing `currentStock` values that are missing or invalid are treated as zero.
- New store items receive `unitRate = totalAmount / totalQty` and `minStock = 0`.
- If `stockUpdated = YES` already exists on the GRN, posting returns `alreadyPosted: true`.

### Inventory item creation
- Store items are created in `lib/api-handlers/purchase-system.ts` with the
  `STK` prefix.
- `currentStock` is set to the received quantity.
- `warehouse` is copied from the GRN.
- `itemCode` is generated the same way as all other system codes.

### Status behavior
- `status` column is always updated to reflect workflow state.
- `GRN` posting sets `status = STOCK_UPDATED`.
- Promotion actions may advance the source document status.

2. Suppliers → New → save → code shows **SUP-0001**, field is locked. Set ACTIVE.
3. Requisition → New → item + qty → save (**PR-0001**). Approve it.

7. GRN preview → **Raise Payment** → save (**PAY-0001**).
8. Inventory → Store → auto-generated `STK-` codes are visible and locked.
